## 1. 背景

### 1.1 业务背景
- 当前场景：约 2000 万活跃用户 × 约 11 个权益方案（洗车券、加油券等），通过 XGB 模型对笛卡尔积打分，每个用户得到一份按 `score` 排序的 `rank=1..N` 推荐列表。
- 离线结果存储于 Hive 表 `dw_jpark.rights_coupon_rec_project_result_json`,导入 HBase 表 `dw_jpark.hb_rights_coupon_rec_project_result_b` 供线上查询(rowkey 为 `REVERSE(user_id)`)。
- AB 实验结论：**洗车券方案 `PC202602050936371715192` 的转化率高于 ECPM 对照组**,而其他方案未观察到稳定优势。

### 1.2 业务诉求
**仅对"模型 rank=1 即为洗车券"的用户进行 XGB 推荐;其余用户走 ECPM。**

## 3. 现状流程 vs. 目标流程

### 3.1 现状流程(改造前)
```
用户请求 → 命中 XGB 流量(flowAbtPlanNo=B)
            ├─ HBase 命中:取 rank=1 plan → 过滤合规? → 合规则推送
            │                                       └ 不合规则取 rank=2 → 顺延...
            │   bizAbtPlanNo = B
            └─ HBase 未命中:走 ECPM
                bizAbtPlanNo = A
```

### 3.2 目标流程(改造后)
```
用户请求 → 命中 XGB 流量(flowAbtPlanNo=B)
            ├─ HBase 命中:取 rank=1 plan
            │     ├─ rank=1 == 洗车券 PC202602050936371715192
            │     │     ├─ 业务过滤通过 → 推送洗车券,bizAbtPlanNo = B
            │     │     └─ 业务过滤拒绝 → 回退 ECPM,bizAbtPlanNo = A
            │     └─ rank=1 != 洗车券
            │           → 回退 ECPM,bizAbtPlanNo = A
            └─ HBase 未命中(冷启/不在预测集) → 走 ECPM(不变)
                bizAbtPlanNo = A
```

> **重要变化**:相比改造前,XGB 分支**不再有"顺延 rank=2/3/..."的逻辑**。本次策略将"只推洗车券"理解为严格语义 —— 一旦 rank=1 不是洗车券、或者 rank=1 洗车券未通过业务过滤,XGB 决策即终止,全部回退 ECPM。原有的多档顺延能力依然保留在 ECPM/FM 分支(若其内部有类似机制),但 XGB 分支只看 rank=1。

### 3.3 ECPM / FM 流量
- A 组(ECPM)、C 组(FM)流量逻辑**完全不变**。本次改造仅影响 B 组(XGB)的决策分支。

## 4. 改造范围(给开发工程师)

### 4.1 需要修改

**模块**:推荐引擎在线服务 - XGB 决策分支(B 流量入口)

**改动点**:解析 HBase 返回的 `recommendation_json` 之后,**仅判定 rank=1**;若不是洗车券 _或_ 未通过业务过滤,均回退 ECPM。**不再顺延 rank=2/3/...**。

```pseudo
plans = parse(recommendation_json).model_1
top1 = plans.find(p => p.rank == 1)          // 注意:按 rank 字段取,不依赖数组下标

if top1 == null || top1.plan_no != CARWASH_PLAN_NO:
    return fallbackToECPM()                  // flowAbtPlanNo=B, bizAbtPlanNo=A

if !businessFilter.pass(user, top1):         // 业务过滤(黑名单/地域等)
    return fallbackToECPM()                  // flowAbtPlanNo=B, bizAbtPlanNo=A

return push(top1)                            // flowAbtPlanNo=B, bizAbtPlanNo=B
```

> **隐含简化**:原 XGB 分支中的"过滤失败后顺延 rank=2/3/..."循环代码,在本次改造中**对 XGB 路径不再生效**。如该顺延逻辑是 XGB 分支独有的代码段,可直接移除;如它是与 ECPM/FM 共用的通用过滤工具方法,**保留方法本体,仅在 XGB 调用处不再走循环**。请开发评估实际代码结构后做最小化改动(参考 AI 编码指南:不要"顺手"重构无关代码)。

### 4.2 保持不变

| 项 | 说明 |
| --- | --- |
| HBase 表结构、rowkey、写入流程 | `hbase_export.sql` 无需修改 |
| ECPM 策略、FM 策略 | A、C 流量分支不动 |
| 业务过滤规则(地域/黑名单等) | 过滤器本体逻辑不变 |
| XGB 用户未命中时的 ECPM 回退 | 冷启动 / 不在预测集用户行为完全保留 |
| 埋点字段 `flowAbtPlanNo` / `bizAbtPlanNo` | 字段语义与赋值规则不变 |
| `recommendation_json` 反序列化与排序假设 | 仍按 `rank` 升序消费,**不要**重新打分 |

### 4.3 业务已确认事项(决策记录)

| # | 问题 | **业务决策** | 实现含义 |
| --- | --- | --- | --- |
| Q1 | 当 rank=1 是洗车券但**被业务规则过滤掉**时,是否顺延 rank=2/3/...? | **不顺延,直接回退 ECPM** | XGB 分支只看 rank=1;rank=1 洗车券一旦过滤拒绝就退出 XGB 决策 |
| Q3 | 回退到 ECPM 时,`bizAbtPlanNo` 是否仍记为 `A`? | **是**(与 HBase 未命中回退一致) | 所有从 XGB 退到 ECPM 的请求统一为 `flowAbtPlanNo=B, bizAbtPlanNo=A` |

### 4.4 仍待业务确认的小项(不阻塞开发)

| # | 问题 | 建议默认 | 影响 |
| --- | --- | --- | --- |
| Q2 | "洗车券 plan_no" 后续是否会扩展为多个? | 当前按单个常量实现,但**配置项使用列表类型**,扩展时只改配置不改代码 | 配置抽象层级 |
| Q4 | 是否需要新增埋点字段区分"回退原因"(未命中 / Top1 非洗车券 / 过滤拒绝)? | 本期**不新增**(埋点字段保持不变);如后续 AB 分析需要归因,单独立项 | AB 数据归因颗粒度 |

### 4.4 流量影响预估(开发与业务都需要知道)

根据 `log_statistics_result_distribution.hql` 中 rank=1 的分布:

| Rank=1 plan_no | 用户数 | 占比 | 改造后处理 |
| --- | ---: | ---: | --- |
| `PC202604301721106437930` | 26,052,443 | 98.0% | **回退 ECPM** |
| `PC202602050936371715192`(洗车券) | 382,256 | **1.4%** | **走 XGB 推洗车券**(若过滤通过) |
| `PC202510141140162778374` | 49,340 | 0.2% | 回退 ECPM |
| 其他 | ~15,372 | <0.1% | 回退 ECPM |
| **合计** | ~26,499,423 | 100% | — |

> **关键结论**:
> 1. 改造后,B 流量中真正由 XGB 推送洗车券的用户占比从~100% 降至 **不超过 ~1.4%**;其余 ≥98.6% 将以 `flowAbtPlanNo=B, bizAbtPlanNo=A` 形式回退到 ECPM。
> 2. 实际的 XGB 推送占比还会被业务过滤(黑名单/地域)进一步削减 —— 即 1.4% 是上限,真实值取决于过滤通过率。
> 3. 这一行为请业务/算法确认,尤其关注:对 AB 报表口径、XGB 模型训练样本回流(曝光/转化样本来源)的潜在影响。

## 5. 测试方案(给测试工程师)

### 5.1 测试目标
1. **功能正确性**:新策略按 §3.2 流程图准确执行。
2. **回归不变性**:ECPM、FM 流量、HBase 未命中分支、业务过滤器等未触改造的链路行为完全一致。
3. **埋点正确性**:`flowAbtPlanNo` / `bizAbtPlanNo` 在各分支下取值符合 §5.3 表格。
4. **性能**:改造仅增加一次字符串比较,P99 延迟不应有可观测劣化。

### 5.2 测试场景矩阵(核心用例)

| 用例编号 | 流量分组 | HBase 是否命中 | rank=1 plan | 业务过滤情况 | 期望推送结果 | `flowAbtPlanNo` | `bizAbtPlanNo` |
| --- | --- | --- | --- | --- | --- | --- | --- |
| TC-01 | XGB(B) | 命中 | 洗车券 | rank=1 通过 | **洗车券**(rank=1) | B | **B** |
| TC-02 | XGB(B) | 命中 | 洗车券 | rank=1 被过滤拒绝 | **ECPM 结果**(不顺延 rank=2) | B | **A** |
| TC-03 | XGB(B) | 命中 | **非**洗车券(如 `PC202604301721106437930`) | 不进过滤(直接回退) | ECPM 结果 | B | A |
| TC-04 | XGB(B) | 命中 | **非**洗车券,但 rank=2/3 中有洗车券 | 不进过滤 | ECPM 结果(只看 rank=1) | B | A |
| TC-05 | XGB(B) | **未命中**(冷启/不在预测集) | — | — | ECPM 结果 | B | A |
| TC-06 | ECPM(A) | — | — | — | ECPM 结果(回归) | A | A |
| TC-07 | FM(C) | — | — | — | FM 结果(回归) | C | C |
| TC-08 | XGB(B) | 命中,JSON 解析异常 | — | — | ECPM 兜底(异常路径) | B | A |
| TC-09 | XGB(B) | 命中,`model_1` 为空数组 | — | — | ECPM 兜底 | B | A |
| TC-10 | XGB(B) | 命中,rank 字段乱序(`rank=2` 排在数组前) | 洗车券为 rank=1 | rank=1 通过 | 洗车券 | B | B |

> **关键回归点**:
> - **TC-02**:验证"不顺延"行为 —— 即使 rank=2 是合规的非洗车券,也**不应**被推送,这是本次改造与原逻辑最大的不同。
> - **TC-04**:验证 XGB 分支**只看 rank=1**,即便后续 rank 有洗车券也不参与决策。
> - **TC-10**:验证按 `rank` 字段判定 top1,而非数组下标。测试数据中至少包含一条 rank 乱序样本。

### 5.3 验收标准(Pass / Fail 判定)

每条用例同时满足以下条件方可 PASS:

1. **推送结果**:HTTP 返回中的方案号(或方案对象)与"期望推送结果"列一致。
2. **埋点字段**:落到日志/Kafka 的 `flowAbtPlanNo`、`bizAbtPlanNo` 与期望值一致。
3. **响应链路**:对于"走 XGB 推洗车券"用例,日志中应能看到 HBase 查询调用记录;对于"回退 ECPM"用例,日志中应能看到 ECPM 调用记录。
4. **性能**:全链路 P99 延迟相对基线增加 ≤ 1ms(改造仅增加一次字符串比较,合理预期为亚毫秒级)。

### 5.4 测试数据准备

#### A. 离线侧(用于灌入 HBase 测试环境)
构造一份小批量(建议 ≥10 条)样例数据,需覆盖:
- rank=1 = `PC202602050936371715192`(洗车券)的用户 ≥2 条(TC-01、TC-02);
- rank=1 = `PC202604301721106437930`(非洗车券,最大占比)的用户 ≥1 条(TC-03);
- rank=1 是非洗车券、但 rank=2/3 中含洗车券的用户 ≥1 条(TC-04);
- rank 字段乱序的用户 ≥1 条(TC-10,洗车券在数组中不排第一,但 rank 字段为 1);
- `model_1` 为空数组的用户 ≥1 条(TC-09);
- `recommendation_json` 非法字符串的用户 ≥1 条(TC-08);
- **不灌入** HBase 的"幽灵用户"≥1 条(TC-05 冷启动)。

#### B. 在线侧
- 业务过滤器配置至少 2 套:
  1. "全通过":用于 TC-01、TC-10 等验证正向路径;
  2. "黑名单包含 `PC202602050936371715192`":用于 **TC-02**(核心:验证洗车券被拒后不顺延);
- 流量分组配置:确保测试账号可被路由到指定 ABT 组(A/B/C)。

### 5.5 上线灰度与监控指标

| 指标 | 上线前基线 | 灰度期预期值 | 告警条件 |
| --- | ---: | --- | --- |
| `flowAbtPlanNo=B & bizAbtPlanNo=B` 占比(B 组内) | ~100% | 预期 **≤1.4%**(实际值 = 1.4% × 业务过滤通过率) | 实际值 >2% 或为 0 |
| `flowAbtPlanNo=B & bizAbtPlanNo=A` 占比(B 组内) | 仅冷启 | 预期 **≥98.6%** | — |
| B 组中 XGB 实际推送方案分布 | 长尾,top1 为 `PC202604301721106437930` | 改造后应**100%** 为 `PC202602050936371715192` | 出现非洗车券 → 立即告警并回滚 |
| 推荐引擎 P99 延迟 | 基线值 | 不应上升 >1ms | 上升 >5ms |
| 洗车券曝光 → 转化率 | AB 实验值 | 应稳定或提升 | 显著下降 → 评估回滚 |

### 5.6 回滚方案
1. 改造代码以**配置开关**(如 `recommend.xgb.carwash_only=true|false`)控制,默认开;
2. 出现异常时改配置回退即可,无需重新发版;
3. HBase 数据不受影响,无数据层回滚成本。

> **强烈建议**开发将该配置项接入配置中心而非 hardcode,作为本次上线的硬性要求。
