# 统一搜索联想词架构 — V4：算法驱动版

## 0. 摘要

V3 架构强调“重离线预计算、轻在线逻辑”，候选池打分在离线 MapReduce 中完成截断，在线仅执行 DAT 查询与状态拼接。该方案性能优异且工程稳定，但排序机制无法利用机器学习自适应迭代，直达产品推荐难以覆盖语义相关但字面不匹配的联想词。

本版本以《口袋APP搜索整体规划》第一阶段“联想词+直达推荐联合优化”及第二阶段“联想搜索区优化”为蓝本。核心改造在于将排序、直达产品路由与覆盖率治理从“离线统计公式 + 人工规则”迁移为“机器学习模型 + 精准/语义双路召回”，并保留前缀/拼音低延迟召回、实时库存双缓冲拼接及冷启动曝光等底层工程能力。

文档结构如下：
*   **第 1 章**：说明规划方向与架构设计的映射关系。
*   **第 2 章**：重构候选召回层，补充语义召回通道。
*   **第 3 章**：联想词排序模型化方案。
*   **第 4 章**：直达产品路由（Onebox）重构与能力扩展。
*   **第 5 章**：覆盖率提升与专项治理机制。
*   **第 6 章**：离线训练与在线 Serving 流水线整合。
*   **第 7 章**：分阶段落地计划。
*   **第 8 章**：版本核心差异总结。

## 2. 候选召回层：多路召回重构

候选池由单一离线排定转变为在线排序模型的输入池，召回通道扩展为三路并行。

### 2.1 精确/前缀召回

保留字符前缀与拼音前缀索引结构。该通道负责在毫秒级预算内圈定初始候选集合，不再承担排序职责。

### 2.2 行为召回

按归一化 Query 聚合会话级日志（包含曝光/点击/转化/去重 Session 数）。其产出不再直接作为候选与综合分数，而是用于生成排序模型的训练样本，并为未被前缀命中的长尾 Query 提供基础召回。

### 2.3 语义召回

复用搜索体系的双塔 DSSM 模型产出 Query 与产品的语义向量。以离线批量方式构建向量索引。当精确召回与行为召回候选不足时，系统使用输入前缀的向量检索最近邻并补入候选池，重点解决长尾覆盖与语义匹配问题。三路候选将统一附带 `source ∈ {前缀, 行为, 语义}` 标签以供模型特征抽取。

## 3. 联想词排序模型化

### 3.1 训练样本构建

基于会话日志，以“曝光未点击”为负样本、“曝光且点击”为正样本，按时间窗口构建 Query 级样本。第二阶段将样本标签扩展为点击与转化双重标签，支持联合建模。

### 3.2 特征工程

*   **Query 侧特征**：历史统计量、Wilson 平滑 CTR、召回来源标签、语义向量。
*   **上下文特征**：请求时段特征、渠道标识及合规允许内的分群特征。
*   **产品关联特征**：若联想词已关联直达产品，引入相关产品的业务位次及质量特征。

### 3.3 模型选型

采用 XGBoost 或 XGBoost-Pairwise 进行排序模型训练，保障在线推理延迟可控。

### 3.4 联合建模演进 (第二阶段)

*   构建多任务学习结构，共享底层特征，分别预估点击率与转化率。
*   最终排序得分根据业务目标（如转化产能）将两项指标加权融合。
*   在合规前提下逐步引入转化后特征进行模型迭代。

### 3.5 在线服务降级

配置明确的延迟阈值。当模型服务异常或超时时，自动降级至基于历史行为统计的 Wilson 平滑 CTR 兜底排序规则。

## 4. 直达产品路由 (Onebox) 重构

### 4.1 存量可直达推荐提升

*   **精准推荐**：维持联想词与产品名称完全匹配时的最高优先级直达路由。
*   **语义推荐**：应用语义召回能力，计算联想词与产品的语义及共现相关性，召回非完全匹配但高相关的产品。
*   **高效排序**：整合召回产品的行为集中度、语义相关性及业务特征，由排序模型决定最终展示次序。

### 4.2 不可直达推荐挖掘

针对原先无直达规则的联想词，通过语义召回与排序模型挖掘高相关候选产品。此类算法挖掘的结果必须经由业务及合规人员复核确认后方可上线，合规审核结果并入最终路由表。

### 4.3 业务域扩展 (第二阶段)

将语义召回与排序模型的应用范围从理财类逐步扩展至信用卡、贷款等非理财 Query 场景。

### 4.4 状态实时拼接

保留本地化双缓冲状态表（`product_id` 寻址）。产品状态秒级更新。命中路由后读取状态快照，判定在售则渲染卡片；若状态异常或数据超时则降级展示普通列表。

## 5. 联想词覆盖率提升

### 5.1 数据缺口归因

系统定期扫描前缀未命中候选池的 Query 数据，按业务场景与拼音变体等维度进行聚类归因。

### 5.2 专项治理

对理财产品类无联想词 Case 设立专项跟踪机制，高频验证召回覆盖率。

### 5.3 语义能力叠加

在冷启动产品目录及金融词典约束的基础上，针对字面未覆盖的长尾 Query，利用向量检索提取语义相关的历史高频 Query 补充至候选池。

### 5.4 冷启动机制强化

新产品沿用业务提权及按天衰减的曝光策略。新候选词生成方式由词典分词升级为基于新品名称与标签向量相似度检索历史 Query。候选词保留来源标签以追踪冷启动效果。

## 6. 流水线整合

### 6.1 离线链路

日志聚合 → 样本构建 → 特征处理 → 模型训练（XGBoost 排序及 DSSM 双塔） → 离线评估验证 → 模型分发。

### 6.2 在线链路

三路并行召回 → 特征数据组装 → 模型实时推理排序 → Onebox 路由判定 → 本地双缓冲状态校验 → 终端渲染。

### 6.3 调度频率

*   **排序模型**：T+1 或高频微调训练。
*   **语义索引**：与主索引 4-6 小时更新周期对齐。
*   **状态表**：秒级并发安全更新。

### 6.4 异常降级

模型链路设置强阻断降级，异常时全量回退至基于规则与统计打分的基线策略。

## 7. 落地计划

*   **第一阶段（Q3，P0）**：联想词排序模型上线；Onebox 语义召回及排序模块投产；开启含人工复核的算法挖掘推荐；启动理财 Query 覆盖率治理。
*   **第二阶段（Q4，P0）**：多目标联合建模投产；直达推荐扩展至全量产品域；全域语义向量基础能力并网。

## 8. 核心差异总结

| 评估维度 | V3 (统计规则体系) | V4 (算法驱动体系) |
| :--- | :--- | :--- |
| **候选召回** | 前缀与行为两路召回，离线排定终局 | 增设语义召回，候选持续输入在线排序阶段 |
| **联想词排序** | 离线计算平滑 CTR 与冷启动分数，固化至 DAT | 部署 XGBoost 模型进行实时特征推理及多任务预估 |
| **直达路由** | 仅依赖规则匹配与统计阈值 | 引入语义召回、模型排序，覆盖全量业务并引入合规复核机制 |
| **覆盖治理** | 被动依赖行为统计积累 | 建立专项数据归因，主动应用语义扩展召回 |
| **状态校验** | 本地双缓冲状态机制 | 沿用该架构确保底线可用性 |
| **高可用策略** | 纯内存查找无依赖 | 模型异常时触发自动降级兜底 |
